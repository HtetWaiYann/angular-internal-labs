Why track matters for list performance in Angular.

What does track do?

In @for (item of items(); track expr), the track expression gives each item a stable identity. Angular uses it to decide: "Is this the same item as before, or a different one?" Same identity → reuse the existing DOM node. Different identity → destroy the old node and create a new one.

track item (object reference)

Using track item uses the object reference as identity. When you refetch data from a server, you get new objects (same data, new refs). Angular sees "new identity" for every item and destroys/recreates all nodes. This is expensive.

track item.id: stable identity

With track item.id, Angular uses a stable identifier that survives refetches. When you get new objects from the server (same ids), Angular reuses existing DOM nodes and updates bindings. No destroy/create. Much cheaper.

When to use trackBy

Use a stable track expression whenever your list can reorder, add, or remove items. Static lists that never change may not need it, but it's a good habit. Use a unique id or a composite key (e.g. track item.id or track item.category + item.id).

Performance impact

Destroying and recreating DOM nodes is expensive: component lifecycle (constructor, ngOnInit, etc.), DOM allocation, and style recalculation. Reusing nodes avoids all of that. In long lists with frequent updates, the difference is dramatic.
