Navigation sequence

Every navigation fires the same sequence of events on Router.events. Subscribe anywhere to observe the full lifecycle.

Event order

  1. NavigationStart
  2. RoutesRecognized
  3. GuardsCheckStart / GuardsCheckEnd
  4. ResolveStart / ResolveEnd
  5. NavigationEnd

Steps 3 and 4 are skipped if the route has no guards or resolvers. A blocked guard emits NavigationCancel instead of NavigationEnd.

Subscribing

Inject Router and subscribe to router.events. Filter by event type using instanceof or the RxJS filter operator.

Guard types

  • canActivate — block entering a route
  • canActivateChild — block entering any child route
  • canDeactivate — block leaving a route
  • canMatch — prevent a route from being matched at all

Return values

  • true — allow navigation
  • false — block navigation (emits NavigationCancel)
  • UrlTree — redirect to a different URL
  • Observable / Promise — async guard; navigation waits

Execution order

canMatch runs first (during route matching). Then canActivate — child-to-parent order. Then canActivateChild. Finally canDeactivate on the route being left.

What resolvers do

Resolvers fetch or compute data before the component activates. The result is placed in ActivatedRouteSnapshot.data under the key you specify. The component receives fully-loaded data on first render — no loading spinner needed in the component.

Trade-off

Resolvers delay navigation until data is ready. A slow resolver blocks the entire navigation. Use the delay picker to see this effect live. For large datasets, prefer fetching in the component and showing a skeleton UI instead.

Accessing data

Inject ActivatedRoute and read route.snapshot.data['key'] for one-time access, or route.data (an Observable) to react to changes when the same component is reused across navigations.

How lazy loading works

When a lazy route is first activated, the router requests the JS chunk from the server. RouteConfigLoadStart fires immediately; RouteConfigLoadEnd fires once the chunk is parsed and ready. The module is then cached for the session.

loadComponent vs loadChildren

  • loadComponent — lazy-loads a single standalone component (one chunk)
  • loadChildren — lazy-loads a route array, useful for feature modules with multiple routes

Preloading

Angular's PreloadAllModules strategy preloads lazy chunks in the background after the app is idle, eliminating the load delay for subsequent visits. Custom preloading strategies give fine-grained control.
