Dependencies

effect()

effect() runs a side-effect when any signal it reads changes. Angular tracks which signals were read during the run and re-runs the effect when any of them change. Runs are scheduled asynchronously (after the current sync block).

Initial run

The effect runs once when the reactive graph is first set up, so you may see one log entry on load. Each click of Increment schedules another run.

Execution log

Each time the effect runs, we log a line. Use this to see exactly when the effect fires in response to signal updates.

} @if (activeTab() === 'computed') {

Dependencies

source → computed → effect

computed()

computed() derives a value from other signals. It recalculates only when one of its dependencies changes, and caches the result until then. Reading it in an effect ties the effect to that computed value.

Recalculation

When the source signal changes, the computed recalculates. The effect that reads the computed then runs, so we log each recompute in the panel.

} @if (activeTab() === 'batching') {

Dependencies

a, b → effect

Batching

When you update multiple signals in the same synchronous block, Angular batches notifications. An effect that reads those signals runs once after the block, not once per update.

Why it matters

Batching avoids redundant effect runs and keeps the reactive graph efficient when many signals change together.

} @if (activeTab() === 'mutation') {

Reference equality

Angular signals notify consumers only when the new value is not equal to the previous one. For objects and arrays, comparison is by reference (===). Mutating in place and returning the same reference does not trigger updates.

Immutable updates

Returning a new array or object (e.g. [...arr, item]) gives a new reference, so the signal notifies and effects/views that read it re-run. Prefer immutable updates for predictable reactivity.
